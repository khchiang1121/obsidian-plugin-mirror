Internal plugin distribution — offline & restricted networks

Obsidian Plugin Mirror

BRAT installs beta Obsidian plugins by talking straight to GitHub. On a vault that can't reach github.com, that's a dead end. This project mirrors plugin releases onto an internal static server, then browses, installs, and updates them from inside the vault — GitHub is never in the loop.

Sheet 01the problem

Same conventions as BRAT, minus the GitHub dependency

The release-asset layout (manifest.json, main.js, optional styles.css / manifest-beta.json) and the semver-with-fallback version sorting are carried over unchanged from BRAT, so plugin authors need to change nothing. What changes is where the installer looks: an internal mirror instead of api.github.com.

Sheet 02system diagram

Two subsystems, one shared contract

mirror-builder and the installer plugin never talk to each other — only through a static file format that nginx serves as-is. Either side can be rebuilt independently as long as it honors that format.

Sheet 03the two subsystems

What each half actually does

mirror-builder

A one-shot Node.js/TypeScript CLI. Reads a tracked-plugin config, fetches releases via the GitHub API with bounded concurrency and an optional token, sorts and retains versions, and writes a spec-compliant static tree — ready to bake into an nginx image with one npm script. Re-runs only fetch what's new and prune what fell out of retention.

  • TypeScript
  • @octokit/rest
  • vitest · 37 tests
mirror-builder/

obsidian-installer-plugin

A standard Obsidian community plugin. Lists everything in the mirror's registry with an Install action, tracks installed mirrored plugins independently of Obsidian's own manifest, and checks for updates on startup and on demand — respecting a per-plugin prerelease opt-in and independently toggleable auto-check / auto-install settings.

  • TypeScript
  • esbuild
  • vitest · 30 tests

Sheet 04quick start

Build the mirror, then install the plugin

01Configure and build the mirror
mirror-builder/
npm install
cp tracked-plugins.example.json tracked-plugins.json   # edit with real owner/repo entries
export GITHUB_TOKEN=ghp_xxx                             # optional, raises the rate limit
npm run build:image                                      # fetch releases, then docker build
02Run it on the internal network
shell
docker run -d --name plugin-mirror -p 8080:80 obsidian-plugin-mirror
03Install the vault plugin
obsidian-installer-plugin/
npm install
npm run build   # typechecks, then bundles src/main.ts -> main.js

Copy manifest.json, main.js, and versions.json into <vault>/.obsidian/plugins/obsidian-mirror-installer/, enable it from Community Plugins, then set the mirror's address in the plugin's settings tab.

Sheet 05known limitations · v1

What's deliberately out of scope for now

LimitationWhy it's fine for now
No file integrity checks Checksums/signatures are noted in the format spec as a future addition — the version-entry format has room for it without a breaking change.
No access control / TLS The mirror is assumed to run on a trusted internal network — an nginx config concern, orthogonal to the file format.
A pruned version can 404 Only affects showing "what version am I on" — the install/update flow itself is unaffected.
