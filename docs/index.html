Obsidian Plugin Mirror

An internal, GitHub-free replacement for BRAT. BRAT installs beta plugins by talking straight to GitHub — on a vault inside a restricted network that can't reach github.com, that's a dead end. This project mirrors plugin releases onto an internally-hosted static server, then browses, installs, and updates them from an Obsidian plugin that only ever talks to that mirror.

TypeScript Docker + nginx Obsidian community plugin Zero GitHub access required inside the vault

Two subsystems, one shared contract

mirror-builder and the installer plugin never talk to each other directly — only through a static file format that nginx serves as-is. Either side can be rebuilt independently as long as it honors that format.

Key features

🔢

Tunable retention

defaultRetain keeps the N newest versions; defaultMinStableRetain guarantees a floor of non-prerelease versions so a burst of betas can't crowd stable releases out entirely.

🩹

Self-healing version tracking

The installer re-reads the real on-disk manifest.json as ground truth on every check, so drift from Obsidian's own updater or a manual copy never causes a stale update prompt.

🔍

Adopts externally-installed plugins

A plugin installed via Obsidian's built-in browser, BRAT, or a manual copy is detected and folded into update checks automatically — no separate "import" step.

🧪

Per-plugin prerelease opt-in

Each tracked plugin independently toggles whether prerelease versions count as an available update.

📴

Zero GitHub dependency inside the vault

The installer plugin only ever calls the internal mirror's HTTP endpoints — index.json and versions.json — never api.github.com.

♻️

Incremental rebuilds

Re-running mirror-builder only fetches what changed and prunes whatever fell out of retention — it never re-downloads everything from scratch.

Non-goals

  • File integrity checks (checksums/signatures) — noted in the format spec as a future addition, not yet implemented
  • Access control or TLS on the mirror itself — assumed to run on a trusted internal network; an nginx config concern, orthogonal to the file format
  • Submitting to Obsidian's official Community Plugins directory — this project exists specifically to work without GitHub access, which the official directory requires

Quick Start

Build the mirror outside the restricted network, then install the vault plugin inside it.

Prerequisites

  • Node.js + npm
  • Docker, to bake the mirror into an nginx image
  • Optionally a GITHUB_TOKEN — raises the GitHub API rate limit from 60 to 5000 requests/hour

Installation

  1. Configure and build the mirror 
    mirror-builder/
    npm install
cp tracked-plugins.example.json tracked-plugins.json   # edit with real owner/repo entries
export GITHUB_TOKEN=ghp_xxx                             # optional, raises the rate limit
npm run build:image                                      # fetch releases, then docker build
  2. Run it on the internal network 
    shell
    docker run -d --name plugin-mirror -p 8080:80 obsidian-plugin-mirror
  3. Build the installer plugin 
    obsidian-installer-plugin/
    npm install
npm run build   # typechecks, then bundles src/main.ts -> main.js
  4. Install the vault plugin

    Copy manifest.json, main.js, and versions.json into <vault>/.obsidian/plugins/obsidian-mirror-installer/, then enable it from Community Plugins.

  5. Point it at your mirror

    Open the plugin's settings tab and set Mirror base URL to your mirror's address, e.g. http://plugin-mirror.internal:8080/.

Tip: Opening the settings tab always runs a fresh update check first — including adopting any plugin already installed by other means — so what you see is current, not stale.

Architecture

Directory structure

tree
obsidian-plugin-mirror/
├── mirror-builder/              # CLI — fetches releases, writes the static tree
│   ├── src/
│   │   ├── cli.ts                 # pipeline entrypoint
│   │   ├── config.ts              # tracked-plugins.json parsing/validation
│   │   ├── github.ts              # @octokit/rest release fetching
│   │   ├── versionSort.ts         # newest-first sort + retention (incl. minStableRetain floor)
│   │   ├── assets.ts              # download + prune stale version directories
│   │   ├── manifestReader.ts      # id/name/author/description extraction
│   │   └── writer.ts              # index.json / versions.json output
│   └── tracked-plugins.json     # the 113-plugin config this site's table reflects
└── obsidian-installer-plugin/   # the vault-side Obsidian plugin
    └── src/
        ├── main.ts                # plugin entrypoint, startup update check
        ├── registry.ts            # index.json / versions.json fetching
        ├── installer.ts           # install/remove + adoptUntrackedInstalledPlugins
        ├── updater.ts             # checkForUpdates
        ├── versionCompare.ts      # semver-with-fallback candidate selection
        └── settingsTab.ts         # Installed / Available UI

mirror-builder pipeline

  1. Load & validate config

    A missing file, invalid JSON, or an empty plugins array is the only fatal error path — it exits non-zero and writes nothing.

  2. Fetch releases per repo

    Bounded concurrency via octokit. A failed fetch for one repo is logged as a warning and skipped — never fatal to the run.

  3. Per-repo processing

    Validate required assets, sort newest-first, apply retain and the minStableRetain floor, then download and prune version directories.

  4. Write output

    index.json covering every plugin that produced at least one valid version, plus a versions.json per plugin.

Installer update-check lifecycle

  1. Fetch the registry index

    Runs on startup (if enabled), on the "Check for updates now" button, and whenever the settings tab opens.

  2. Adopt untracked installed plugins

    Any registry entry with a matching .obsidian/plugins/<id>/manifest.json on disk that isn't already tracked gets folded in, using the real on-disk version as ground truth.

  3. Per-plugin version check

    Compares the installed version against the newest eligible version (respecting each plugin's prerelease opt-in).

  4. Optionally auto-install

    If "Install updates automatically" is on, eligible updates are downloaded and applied immediately.

  5. Persist settings

    Saved unconditionally — this is also where a self-healed stale version or a newly-adopted plugin gets written to disk.

Shared Format

The only contract between mirror-builder and the installer plugin — two JSON files nginx serves as plain static content, no custom server logic involved.

index.json

One entry per plugin that has at least one valid mirrored version.

id / nameFrom the plugin's own manifest.json
authorFrom manifest.json; falls back to the GitHub repo owner if the manifest omits it
descriptionFrom manifest.json
repoowner/name — the tracked GitHub repo
latestVersionNewest retained non-prerelease version, or null
latestPrereleaseNewest retained prerelease version, or null

versions.json

One per plugin, at plugins/<owner>/<repo>/versions.json.

repo / latestRepo string and the newest non-prerelease version
versions[]Every retained version — version, prerelease, publishedAt, files

Note: Checksums/signatures aren't part of the format yet, but the versions[] entry shape has room for them without a breaking change — see the full spec for details.

Known Limitations

What's deliberately out of scope for v1.

LimitationWhy it's fine for now
No file integrity checks Checksums/signatures are noted in the format spec as a future addition — the version-entry format has room for it without a breaking change.
No access control / TLS The mirror is assumed to run on a trusted internal network — an nginx config concern, orthogonal to the file format.
A pruned version can 404 Only affects showing "what version am I on" — the install/update flow itself is unaffected.

Changelog

obsidian-installer-plugin v1.0.4

Installed/available name consistency

  • Fix The "Installed" list showed the raw plugin id while "Available" showed the registry's human-readable name for the same plugin. TrackedPlugin now stores and self-heals a name field, backfilled for plugins tracked before this fix.
obsidian-installer-plugin v1.0.3

Settings tab render ordering

  • Fix On a fresh launch, the settings tab could render the installed list before the update check finished, so the "Install update" button just wasn't there yet on first paint. It now awaits the check before rendering, with a loading placeholder shown meanwhile.
obsidian-installer-plugin v1.0.2

Startup adoption + update check

  • Fix Adoption of externally-installed plugins previously only ran when the settings tab was opened. checkForUpdates now adopts on every trigger path — startup, the manual button, and the command palette entry.
obsidian-installer-plugin v1.0.1

Manifest author fallback

  • Fix A manifest.json missing the optional author field (not required by Obsidian to load a plugin) no longer excludes that plugin from index.json entirely — falls back to the GitHub repo owner.
obsidian-installer-plugin v1.0.0

Initial release

  • Feature Browse, install, update, and remove plugins from the mirror's registry — including a per-plugin prerelease opt-in and independently toggleable auto-check / auto-install settings.
mirror-builder

Ongoing fixes

  • Feature defaultMinStableRetain — a floor that guarantees N non-prerelease versions are always retained, even if a burst of prereleases would otherwise crowd them out of the plain retain window.
  • Fix A malformed manifest.json on one plugin no longer crashes the whole run — that plugin is skipped with a warning instead.
  • Fix Stale entries are removed from tracked-plugins.json automatically instead of accumulating.
