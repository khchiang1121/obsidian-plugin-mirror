Obsidian Plugin Mirror

An internal, GitHub-free replacement for BRAT. BRAT installs beta plugins by talking straight to GitHub — on a vault inside a restricted network that can't reach github.com, that's a dead end. This project mirrors plugin releases onto an internally-hosted static server, then browses, installs, and updates them from an Obsidian plugin that only ever talks to that mirror. 一個不需要 GitHub 存取權限的內部替代方案，取代 BRAT。BRAT 透過直接連線 GitHub 來安裝測試版外掛——但在無法連到 github.com 的受限網路內，這條路走不通。本專案將外掛的發布版本鏡像到內部託管的靜態伺服器上，再透過一個只會與該鏡像伺服器通訊的 Obsidian 外掛來瀏覽、安裝與更新它們。

TypeScript Docker + nginx Obsidian community pluginObsidian 社群外掛 Zero GitHub access required inside the vault保管庫內完全不需要存取 GitHub

Two subsystems, one shared contract兩個子系統，一份共用契約

mirror-builder and the installer plugin never talk to each other directly — only through a static file format that nginx serves as-is. Either side can be rebuilt independently as long as it honors that format. mirror-builder 與 installer 外掛從不直接互相通訊——只透過 nginx 原樣提供的靜態檔案格式溝通。只要遵守這個格式，任一方都能獨立重建。

Key features主要功能

🔢

Tunable retention可調整的保留策略

defaultRetain keeps the N newest versions; defaultMinStableRetain guarantees a floor of non-prerelease versions so a burst of betas can't crowd stable releases out entirely. defaultRetain 保留最新的 N 個版本；defaultMinStableRetain 則保證一個非預發布版本的底線數量，讓一波測試版不會把正式版全部擠出保留範圍。

🩹

Self-healing version tracking自我修復的版本追蹤

The installer re-reads the real on-disk manifest.json as ground truth on every check, so drift from Obsidian's own updater or a manual copy never causes a stale update prompt. 安裝器在每次檢查時都會重新讀取磁碟上真實的 manifest.json 作為依據，因此不論是 Obsidian 自己的更新機制還是手動複製造成的版本落差，都不會產生過期的更新提示。

🔍

Adopts externally-installed plugins自動採納外部安裝的外掛

A plugin installed via Obsidian's built-in browser, BRAT, or a manual copy is detected and folded into update checks automatically — no separate "import" step. 透過 Obsidian 內建瀏覽器、BRAT 或手動複製安裝的外掛，會被自動偵測並納入更新檢查——不需要另外的「匯入」步驟。

🧪

Per-plugin prerelease opt-in逐外掛設定是否允許預發布版本

Each tracked plugin independently toggles whether prerelease versions count as an available update. 每個追蹤的外掛都能獨立切換是否將預發布版本視為可用更新。

📴

Zero GitHub dependency inside the vault保管庫內零 GitHub 依賴

The installer plugin only ever calls the internal mirror's HTTP endpoints — index.json and versions.json — never api.github.com. installer 外掛只會呼叫內部鏡像伺服器的 HTTP 端點——index.json 與 versions.json——絕不會呼叫 api.github.com。

♻️

Incremental rebuilds漸進式重建

Re-running mirror-builder only fetches what changed and prunes whatever fell out of retention — it never re-downloads everything from scratch. 重新執行 mirror-builder 只會抓取有變動的部分，並清除超出保留範圍的版本——絕不會從頭重新下載所有內容。

Non-goals非目標

  • File integrity checks (checksums/signatures) — noted in the format spec as a future addition, not yet implemented檔案完整性檢查（校驗碼／簽章）——格式規格中已註明為未來增補項目，目前尚未實作
  • Access control or TLS on the mirror itself — assumed to run on a trusted internal network; an nginx config concern, orthogonal to the file format鏡像伺服器本身的存取控制或 TLS——假設執行於受信任的內部網路；這屬於 nginx 設定範疇，與檔案格式無關
  • Submitting to Obsidian's official Community Plugins directory — this project exists specifically to work without GitHub access, which the official directory requires提交到 Obsidian 官方社群外掛目錄——本專案的存在目的正是為了在沒有 GitHub 存取權限的情況下運作，而官方目錄恰好需要它

Quick Start快速開始

Build the mirror outside the restricted network, then install the vault plugin inside it. 在受限網路之外建置鏡像伺服器，再到網路內部安裝保管庫外掛。

Prerequisites先決條件

  • Node.js + npm
  • Docker, to bake the mirror into an nginx imageDocker，用來把鏡像伺服器打包成 nginx 映像檔
  • Optionally a GITHUB_TOKEN — raises the GitHub API rate limit from 60 to 5000 requests/hour選用 GITHUB_TOKEN——可將 GitHub API 速率限制從每小時 60 次提高到 5000 次

Installation安裝步驟

  1. Configure and build the mirror設定並建置鏡像伺服器 
    mirror-builder/
    npm install
cp tracked-plugins.example.json tracked-plugins.json   # edit with real owner/repo entries# 編輯為實際的 owner/repo 項目
export GITHUB_TOKEN=ghp_xxx                             # optional, raises the rate limit# 選用，可提高速率限制
npm run build:image                                      # fetch releases, then docker build# 抓取發布版本，再執行 docker build
  2. Run it on the internal network在內部網路上執行 
    shell
    docker run -d --name plugin-mirror -p 8080:80 obsidian-plugin-mirror
  3. Build the installer plugin建置 installer 外掛 
    obsidian-installer-plugin/
    npm install
npm run build   # typechecks, then bundles src/main.ts -> main.js# 先型別檢查，再將 src/main.ts 打包成 main.js
  4. Install the vault plugin安裝保管庫外掛

    Copy manifest.json, main.js, and versions.json into <vault>/.obsidian/plugins/obsidian-mirror-installer/, then enable it from Community Plugins. 將 manifest.json、main.js 與 versions.json 複製到 <vault>/.obsidian/plugins/obsidian-mirror-installer/，然後在社群外掛中啟用它。

  5. Point it at your mirror指向你的鏡像伺服器

    Open the plugin's settings tab and set Mirror base URL to your mirror's address, e.g. http://plugin-mirror.internal:8080/. 開啟外掛的設定頁籤，將鏡像伺服器網址設為你的鏡像伺服器位址，例如 http://plugin-mirror.internal:8080/。

Tip: Opening the settings tab always runs a fresh update check first — including adopting any plugin already installed by other means — so what you see is current, not stale. 提示：開啟設定頁籤時一律會先執行一次全新的更新檢查——包括採納任何以其他方式已安裝的外掛——因此你看到的一定是最新狀態，而非過期資料。

mirror-builder pipelinemirror-builder 流程

  1. Load & validate config載入並驗證設定

    A missing file, invalid JSON, or an empty plugins array is the only fatal error path — it exits non-zero and writes nothing.檔案不存在、JSON 格式錯誤，或 plugins 陣列為空，是唯一的致命錯誤路徑——會以非零狀態退出且不寫入任何內容。

  2. Fetch releases per repo逐一抓取各 repo 的發布版本

    Bounded concurrency via octokit. A failed fetch for one repo is logged as a warning and skipped — never fatal to the run.透過 octokit 限制並行數量。單一 repo 抓取失敗只會記錄警告並略過——絕不會讓整次執行失敗。

  3. Per-repo processing逐一處理每個 repo

    Validate required assets, sort newest-first, apply retain and the minStableRetain floor, then download and prune version directories.驗證必要的資產檔案、由新到舊排序、套用 retain 與 minStableRetain 底線，接著下載並清除版本目錄。

  4. Write output寫出結果

    index.json covering every plugin that produced at least one valid version, plus a versions.json per plugin.產出涵蓋每個至少有一個有效版本的外掛的 index.json，以及每個外掛各自的 versions.json。

Installer update-check lifecycleInstaller 更新檢查生命週期

  1. Fetch the registry index抓取登錄索引

    Runs on startup (if enabled), on the "Check for updates now" button, and whenever the settings tab opens.在啟動時（若已啟用）、按下「立即檢查更新」按鈕時，以及每次開啟設定頁籤時執行。

  2. Adopt untracked installed plugins採納尚未追蹤的已安裝外掛

    Any registry entry with a matching .obsidian/plugins/<id>/manifest.json on disk that isn't already tracked gets folded in, using the real on-disk version as ground truth.任何在磁碟上有對應 .obsidian/plugins/<id>/manifest.json、但尚未被追蹤的登錄項目，都會被納入追蹤，並以磁碟上的實際版本作為依據。

  3. Per-plugin version check逐外掛版本檢查

    Compares the installed version against the newest eligible version (respecting each plugin's prerelease opt-in).將已安裝版本與最新的合格版本比較（會遵守每個外掛各自的預發布版本設定）。

  4. Optionally auto-install視情況自動安裝

    If "Install updates automatically" is on, eligible updates are downloaded and applied immediately.若開啟「自動安裝更新」，符合資格的更新會立即下載並套用。

  5. Persist settings持久化設定

    Saved unconditionally — this is also where a self-healed stale version or a newly-adopted plugin gets written to disk.無條件儲存——自我修復的過期版本或新採納的外掛，也是在這一步被寫入磁碟。

Shared Format共用格式

The only contract between mirror-builder and the installer plugin — two JSON files nginx serves as plain static content, no custom server logic involved. mirror-builder 與 installer 外掛之間唯一的契約——兩個由 nginx 以純靜態內容提供的 JSON 檔案，不涉及任何自訂伺服器邏輯。

index.json

One entry per plugin that has at least one valid mirrored version. 每個至少有一個有效鏡像版本的外掛，都有一筆對應項目。

id / nameFrom the plugin's own manifest.json來自外掛自身的 manifest.json
authorFrom manifest.json; falls back to the GitHub repo owner if the manifest omits it來自 manifest.json；若 manifest 未提供，則回退為 GitHub repo 擁有者
descriptionFrom manifest.json來自 manifest.json
repoowner/name — the tracked GitHub repoowner/name —— 被追蹤的 GitHub repo
latestVersionNewest retained non-prerelease version, or null保留範圍內最新的正式版本，或 null
latestPrereleaseNewest retained prerelease version, or null保留範圍內最新的預發布版本，或 null

versions.json

One per plugin, at plugins/<owner>/<repo>/versions.json. 每個外掛一份，位於 plugins/<owner>/<repo>/versions.json。

repo / latestRepo string and the newest non-prerelease versionRepo 字串與最新的正式版本
versions[]Every retained version — version, prerelease, publishedAt, files每個保留的版本——version、prerelease、publishedAt、files

Note: Checksums/signatures aren't part of the format yet, but the versions[] entry shape has room for them without a breaking change — see the full spec for details. 備註：校驗碼／簽章目前還不是格式的一部分，但 versions[] 項目的結構已預留空間，未來加入時不會造成破壞性變更——詳見完整規格。
