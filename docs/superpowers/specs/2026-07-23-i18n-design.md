# obsidian-installer-plugin i18n Design

**Goal:** Support English and Traditional Chinese (Taiwan) as display languages for `obsidian-installer-plugin`'s UI, auto-detected from Obsidian's own language setting.

## Scope

In scope:
- All user-facing strings in `src/settingsTab.ts` (setting names/descriptions, button text, placeholders, tooltips, static list-state text).
- All user-facing strings in `src/main.ts` (the one registered command's name, all `Notice` text).

Out of scope:
- `manifest.json`'s `description` field — static metadata Obsidian reads outside our rendering; not worth localizing for one description line.
- `docs/index.html` (the project's docs site) — separate deliverable, not part of the running plugin.
- Live language switching while the plugin is running — locale is detected once at `onload()`, matching Obsidian's own behavior (Obsidian itself needs a reload to fully re-render a language change).

## Locale detection

Obsidian exposes its own configured display language via `window.localStorage.getItem('language')`, returning codes like `'en'`, `'zh'`, `'zh-TW'`, `'ja'`, etc. This project ships exactly one Chinese variant (Traditional, Taiwan), so:

- `'zh'`, `'zh-TW'`, `'zh-CN'`, `'zh-HK'` → `'zh-TW'` locale (Traditional Chinese is still far more useful to any Chinese-locale reader than English, even if their Obsidian is set to Simplified).
- Anything else, including `null`/unset → `'en'`.

Detection happens once, in `main.ts#onload()`, before the settings tab or command is registered, via `setLocale(detectLocale(window.localStorage.getItem('language')))`.

## Module: `src/i18n/`

```
src/i18n/
  locales/
    en.ts        // Record<string, string>
    zh-TW.ts      // same key set, Traditional Chinese values
  index.ts        // detectLocale, setLocale, t
```

`index.ts` exports:
- `type Locale = 'en' | 'zh-TW'`
- `detectLocale(raw: string | null): Locale` — pure function, the mapping table above.
- `setLocale(locale: Locale): void` — sets module-level current locale (defaults to `'en'` before first call).
- `t(key: string, vars?: Record<string, string | number>): string` — looks up `key` in the current locale's dictionary; if missing, falls back to the English dictionary; if missing there too, returns the key itself (never throws). Does `{placeholder}` substitution against `vars`.

Dictionary keys are namespaced by area, e.g.:
- `settings.mirrorBaseUrl.name`, `settings.mirrorBaseUrl.desc`
- `settings.autoCheckOnStartup.name`
- `settings.autoInstallUpdates.name`, `settings.autoInstallUpdates.desc`
- `settings.checkNow.name`, `settings.checkNow.button`
- `installed.heading`, `installed.checking`, `installed.empty`, `installed.noMatch`, `installed.searchPlaceholder`, `installed.status.upToDate`, `installed.status.updateAvailable`, `installed.button.installUpdate`, `installed.button.remove`, `installed.toggle.allowPrerelease`
- `registry.heading`, `registry.loading`, `registry.loadError`, `registry.empty`, `registry.noMatch`, `registry.searchPlaceholder`, `registry.desc`, `registry.button.install`, `registry.pickVersionLink`, `registry.versionPicker.loading`, `registry.versionPicker.loadError`, `registry.versionPicker.empty`, `registry.versionPicker.installButton`
- `self.name`, `self.status.checking`, `self.status.notInRegistry`, `self.status.error`, `self.status.upToDate`, `self.status.updateAvailable`, `self.button.update`
- `notice.updated`, `notice.updateFailed`, `notice.removed`, `notice.removeFailed`, `notice.installed`, `notice.installFailed`, `notice.noInstallableVersion`, `notice.selfUpdated`, `notice.selfUpdateFailed`, `notice.autoUpdated`, `notice.autoUpdateFailed`, `notice.checkFailed`
- `command.checkForUpdates`

All strings with dynamic parts (plugin id, version numbers, counts, error messages) use `{placeholder}` interpolation, e.g. `t('notice.updated', { id, version })` against `'Updated {id} to v{version}'`.

## Call-site changes

`settingsTab.ts` and `main.ts`: every literal string passed to `.setName()`, `.setDesc()`, `.setButtonText()`, `.setPlaceholder()`, `.setTooltip()`, `createEl({ text })`, `new Notice(...)`, and the one `addCommand({ name })` is replaced with a `t(...)` call. No structural/DOM changes — this is a string-replacement pass over existing render logic.

## Testing

`test/i18n.test.ts`:
- `detectLocale` mapping table (each Obsidian language code above, plus `null` and an unrecognized code like `'ja'`).
- `t()` interpolation with vars, `t()` with no vars, `t()` fallback to English on a locale-missing key, `t()` returning the key itself when missing from both dictionaries.
- A parity test asserting `Object.keys(en)` and `Object.keys(zhTW)` are set-equal, to prevent future key drift between the two dictionaries.

No new tests for `settingsTab.ts`/`main.ts` themselves — consistent with this codebase's existing pattern of unit-testing logic modules only, not the DOM-heavy settings tab.

## Verification

- `npm test` (vitest) — new `i18n.test.ts` plus full existing suite unaffected.
- `npm run typecheck`
- `npm run build`
- Manual smoke check not possible without a running Obsidian instance (same limitation noted for this plugin's UI work throughout this project) — call-site correctness verified by code review of every replaced string against the original English text.
